Stable builds & improved definition API

The main focus of Modulo Alpha development at this point is overhauling the API to make custom component parts. This is because the current internal API had inconsistencies and limitations, and it was clearly the absolute central way to extend the framework, and thus very important to get correct before beta. I'm happy to say that the bulk of the internal and CPart API is stable, and the remaining bugs only involve documenting, a few new features and tweaks, and fixing any straggling nested errors.

A quick overview of the new API features:

  1. register() API, which is a "plug-in" system, an easy way to attach (or override) classes and functions to the Modulo object
  2. Definitions API, which is a complete rewrite of the internal loading API to be more transparent and data-oriented for debugging, while producing better builds
  3. Config API, which is a powerful configuration system for both application configuration and 3rd party module configuration.
  4. Processor API which allows for static functions to iterate on definitions, either private to definition classes (as static methods), or as public, re-usable mixins (which is how -src= is implemented)

Part of the reason I had to rewrite it this way was to support adding the following features:

  1. Stable builds with fully bundled self-contained JS, CSS, and HTML (from the browser or CLI)
  2. CPart feature improvements, including State allowing for Stores (globally connected data)
